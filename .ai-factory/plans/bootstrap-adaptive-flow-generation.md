## Architectural Decisions Locked Before Implementation

These are decisions taken at planning time so `/aif-implement` does not re-litigate them per task. Verified against actual code at planning time.

1. **Bootstrap-to-pipeline transition (codified in ADR 0003 — Task 1): two-graph model with a follow-up run.**
   - The bootstrap is its own `Graph` value bundled as `bootstrap.toml` in `crates/surge-core/bundled/flows/`.
   - Three Agent nodes (Description Author → Roadmap Planner → Flow Generator), each with a `HumanGate` after it.
   - Flow Generator's terminal outcome (`approve`) emits `PipelineMaterialized { graph, graph_hash }` into the bootstrap run's event log.
   - The daemon (or in-process driver) detects `PipelineMaterialized` post-`RunStarted` and spawns a **follow-up run** with the materialized graph. The bootstrap run reaches a `Terminal` node and completes normally.
   - The follow-up run reads bootstrap artifacts (`description.md`, `roadmap.md`, `flow.toml`) via the artifact store keyed by `bootstrap_run_id`.
   - Rationale: keeps each run's `PipelineMaterialized` emission atomic with `RunStarted` (preserves the M6 invariant in `engine.rs:190-215`), keeps fold/replay deterministic, and matches the roadmap's "outer milestone Loop wrapping inner task Loop" pattern naturally — the inner pipeline is just another run.
2. **Bootstrap event flow:** Bootstrap-specific events (`BootstrapStageStarted`, `BootstrapArtifactProduced`, `BootstrapApprovalRequested`, `BootstrapApprovalDecided`, `BootstrapEditRequested`) are emitted **alongside** the generic stage events (`StageEntered`, `OutcomeReported`, `ArtifactProduced`) — they are bootstrap **annotations**, not replacements. This keeps generic graph machinery (frames, fold, replay) unchanged.
3. **HumanGate Edit decision (Task 7):** extend `HumanGateConfig` with an opt-in `mode: HumanGateMode { Generic, Bootstrap { stage } }` field carrying `#[serde(default)]` so existing TOML round-trips. In `Bootstrap` mode the gate accepts three outcomes (`approve`, `edit`, `reject`) and emits the `Bootstrap*` events in addition to the generic ones. The `Edit` outcome routes back to the same Agent node via a `Backtrack` edge (declared in `surge-core/src/edge.rs:22-28` but **runtime semantics not yet wired** — see Decision 16). `feedback` (operator-supplied free-text) is stored on `BootstrapEditRequested.feedback` and surfaced to the agent via a new variant `ArtifactSource::EditFeedback { from_node: NodeKey }` on the existing `ArtifactSource` enum in `crates/surge-core/src/agent_config.rs` (NOT a new `BindingSource` enum — none exists in the codebase; bindings flow through `ArtifactSource`).
4. **Edit-loop cap (Task 9):** new `bootstrap.edit_loop_cap` field on `EngineRunConfig` (default `3`). After N rejections of the **same** stage, the gate handler emits `EscalationRequested { stage }` and the run ends with a non-zero terminal — operator must explicitly restart. Counter is per-stage, persisted in `RunMemory.bootstrap_edit_counts: BTreeMap<BootstrapStage, u32>` (deterministic fold).
5. **Validation retry on Flow Generator output (Task 10):** when the `flow.toml` produced by Flow Generator fails graph validation (`validate_for_m6`), the engine emits `BootstrapEditRequested { stage: Flow, feedback: <validation errors as text> }` instead of failing the run, and routes back to Flow Generator via the same `Backtrack` mechanism. Counts toward the same edit-loop cap. Validator errors must be human-readable strings (already the case via `EngineError::Display`).
6. **Archetype detection (Task 11):** Flow Generator chooses the archetype as part of its agent output. We do **not** add a dedicated archetype-detection node. The agent's profile prompt enumerates the seven archetypes (linear-3, linear-with-Review, multi-milestone outer-loop, bug-fix, refactor, spike, single-task) and the agent commits to one in its output via a new top-level `[archetype]` block in the produced `flow.toml`: `archetype = { name = "multi-milestone", milestones = N, edit_loop_cap = ... }`. The block is parsed by `surge-core` and stored on `Graph.metadata.archetype: Option<ArchetypeMetadata>`. Telemetry (Task 24) reads this back.
7. **`--template=<name>` skip path (Task 14):** new `surge engine run --template <name>` shortcut loads a bundled archetype from `crates/surge-core/bundled/flows/` (which now includes the existing `examples/flow_*.toml` files copied with provenance), bypassing bootstrap entirely. Lookup is by name (`linear-3`, `bug-fix`, etc.) with `ArchetypeRegistry` mirroring `ProfileRegistry`. User-supplied templates discovered under `~/.surge/templates/<name>.toml` win over bundled (3-way lookup matches profile registry pattern from M7).
8. **Artifact storage (Task 16):** artifacts produced inside bootstrap (or any run) get persisted to `~/.surge/runs/<run_id>/artifacts/<content_hash>` with a `name` index file. `ArtifactRef.path` (already on the type per `surge-core/run_state.rs:76-91`) is canonicalized to point at this content-addressed location. Resolution helper `ArtifactStore::open(run_id, hash)` lives in `surge-persistence` (NOT in `surge-core` — it does I/O). Bootstrap follow-up run reads via the same helper, parameterized by `bootstrap_run_id` from `EngineRunConfig.bootstrap_parent: Option<RunId>`.
9. **Roadmap-driven outer loop (Task 18):** Flow Generator's multi-milestone archetype emits a graph where the outer node is `Loop` over `IterableSource::Artifact("roadmap.milestones")` with body subgraph holding the inner task `Loop`. The `Loop` and `Subgraph` handlers already exist (per `engine/stage/loop_stage.rs` and `subgraph_stage.rs`). No new node-kind variants. We add a profile prompt instruction telling Flow Generator to wire HumanGates between milestones if `bootstrap.roadmap.gate_between_milestones = true` (default for multi-milestone archetype).
10. **Profile prompts produce structured output (Task 12):** Description Author / Roadmap Planner / Flow Generator profiles all use `report_stage_outcome` to emit a single artifact. Their prompts must reference `{{user_prompt}}`, `{{description_artifact}}`, `{{roadmap_artifact}}` template variables (rendered via the existing `PromptRenderer` from M7). Bindings declared in the bootstrap graph wire them up.
11. **CLI surface (Task 13):** new `surge bootstrap` subcommand group:
    - `surge bootstrap "<free-form prompt>"` — start a bootstrap run, blocking, prints status to stderr, opens HumanGate prompts via the existing console approval channel (M5 baseline).
    - `surge bootstrap resume <run_id>` — resume a bootstrap run that was paused (e.g., daemon mode where Telegram approvals come later).
    - The existing `surge engine run --template <name>` flag (Task 14) is the explicit "skip bootstrap" path.
12. **Telemetry (Task 24):** new event `BootstrapTelemetry { stage, archetype: Option<ArchetypeMetadata>, time_per_stage_ms: BTreeMap<BootstrapStage, u64>, edit_counts: BTreeMap<BootstrapStage, u32> }` emitted at bootstrap completion. Lives in `run_event.rs`, deterministic (no wall-clock reads in fold), times computed from event log seqs and timestamps already on each event.
13. **Logging targets** follow the existing `engine::*` / `profile::*` / `cli::*` naming. Bootstrap-specific code uses `engine::bootstrap` and `engine::bootstrap::stage::{description,roadmap,flow}`.
14. **Errors:** `BootstrapError` family added to `SurgeError` (`MissingBootstrapStage`, `EditLoopCapExceeded`, `ValidationRetryExhausted`, `MaterializedGraphInvalid`, `BootstrapArchetypeMissing`, `BootstrapArchetypeMismatch`). All retrofit `#[non_exhaustive]`.
15. **Clippy budget:** `clippy.toml` enforces cognitive-complexity ≤ 25, function length ≤ 100 lines, max 3 bool struct fields, max 7 fn args (3 bool args max). New modules respect these limits — break long handlers into helpers if needed.
16. **`EdgeKind::Backtrack` runtime semantics (Task 27):** the variant exists in `surge-core/src/edge.rs:22-28` but `crates/surge-orchestrator/src/engine/routing.rs` currently ignores `kind` entirely — every edge is treated as `Forward`. Bootstrap's edit-loop relies on routing back from a HumanGate to the originating Agent node via `Backtrack`. We wire it explicitly: when traversing a `Backtrack` edge, the engine (a) increments the per-node visit counter, (b) does NOT clear already-folded outcomes from `RunMemory` (replay determinism — folds are append-only), (c) re-enters the Agent stage which re-resolves bindings (now including the new `BootstrapEditRequested` feedback). The engine also relaxes the validator (Task 28): a cycle is permitted iff at least one edge in the cycle is `Backtrack` — pure-Forward cycles remain forbidden.
17. **Initial-prompt binding (Task 29):** `RunStarted.initial_prompt` is captured today but never exposed as an artifact. We add `ArtifactSource::InitialPrompt` and have `Engine::start_run` synthesize a `RunMemory.artifacts["user_prompt"]` entry from `RunStarted.initial_prompt` so the bootstrap profiles' `{{user_prompt}}` Handlebars variable resolves through the standard binding path (no special prompt-injection plumbing).
18. **Artifact event upstream (Task 30):** the agent stage's `TODO(M5)` at `crates/surge-orchestrator/src/engine/stage/agent.rs:94-100` must be closed before the artifact store (Task 16) is useful. When a `BridgeEvent::OutcomeReported` arrives with non-empty `artifacts_produced`, the engine emits one `EventPayload::ArtifactProduced { node, artifact, path, name }` per path, computing `ContentHash` from the file contents at the worktree path. This is the upstream that `ArtifactStore::put` consumes.
