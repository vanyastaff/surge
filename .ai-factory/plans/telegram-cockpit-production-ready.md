Rationale: bootstrap & adaptive flow generation is already complete and emits `BootstrapApprovalRequested` / `BootstrapApprovalDecided` / `BootstrapEditRequested` / `PipelineMaterialized` events into the run log (verified in `crates/surge-orchestrator/src/engine/stage/human_gate.rs:74-99`). The TelegramDeliverer currently sends **plain text** (`crates/surge-notify/src/telegram.rs:42-78`) and inbox cards have an *inline-keyboard formatter* (`format_inbox_card` lines 134-177) but the formatter is **not wired into the deliverer**. This milestone closes the loop: bootstrap approvals (and every other `HumanInputRequested`) emit Telegram cards, the operator taps a button, the daemon translates the callback to `Engine::resolve_human_input` (`engine/engine.rs:683`) — the **same** resolution entry point the console-approval path uses. The roadmap line "Sandbox elevation card" is **explicitly deferred** in this plan (see § Out of Scope) — the sandbox-delegation matrix milestone owns that surface. We wire only the events that exist today; sandbox elevation events do not.

## Architectural Decisions Locked Before Implementation

These are decisions taken at planning time so `/aif-implement` does not re-litigate them per task. Verified against actual code at planning time.

1. **New crate `surge-telegram` (NOT a fattening of `surge-notify`).** `surge-notify` stays a *transport library* whose only Telegram surface is `TelegramDeliverer + TelegramSecretResolver + format_inbox_card + InboxCardRendered + InboxKeyboardButton`. The cockpit (long-poll loop, callback router, command handlers, card-state store, rate-limiter, pairing flow) lives in a new crate `surge-telegram` that depends on `surge-notify` for primitives and on `surge-core` / `surge-persistence` / `surge-orchestrator` for types & engine handles. Rationale: `surge-notify` is consumed by `surge-orchestrator` (event-driven push) — pulling teloxide / a long-poll runtime into `surge-notify` would force every consumer (including `surge-cli`'s short-lived commands) to link the bot loop. Separation also matches the existing `surge-daemon::inbox` pattern.

2. **Bot framework: `teloxide` 0.13 (long-poll default).** Roadmap explicitly mandates `teloxide` long-poll as default (line 180). Webhook receiver via `tiny_http` is **deferred** to a follow-up — out of scope here (kept on a deliberate single-channel implementation). `teloxide` is added as a workspace dep with the `macros, ctrlc_handler` features. `wiremock` (already in `workspace.deps`) mocks the Bot API for integration tests by setting `teloxide::Bot::api_url(<mock url>)`.

3. **Reject the `HumanInputResolver` trait abstraction.** The console-approval path in `surge-cli` and the future Telegram-callback path in `surge-telegram` both already call `Engine::resolve_human_input(run_id, call_id, response_json)` directly. Introducing a `trait HumanInputResolver { async fn resolve(..) }` would (a) require an `Arc<dyn HumanInputResolver>` on `Engine` purely for indirection, (b) add a second seam to mock in integration tests, (c) produce zero behavioral difference today. The engine surface is **already the contract** — both channels are clients of it. Documented in ADR 0009 (Task 2) so future contributors don't re-litigate. (If we ever add a third channel — e.g. Slack — the trait becomes worthwhile; today YAGNI applies.)

4. **Card-id correlation key = `(run_id, node_key, attempt_index)`.** This triple is what we hash to derive the SQLite primary key for the new `telegram_cards` table. `attempt_index` is required because the bootstrap edit loop can re-enter the same `(run_id, node_key)` and we need each attempt to land on a fresh card (else `editMessageText` would silently overwrite the previous attempt). `attempt_index` is read from `RunMemory.node_visits[node_key]` (added in Task 27 of the bootstrap milestone — already shipped per `surge-core/src/run_state.rs`). Idempotency is enforced by `INSERT OR IGNORE` on this triple.

5. **`callback_data` schema:** `cockpit:<verb>:<card_id>` where `<card_id>` is the ULID primary key of the `telegram_cards` row (NOT a token, NOT a base64 payload — Telegram caps `callback_data` at 64 bytes and we want the parser to be obvious from the wire). `<verb>` ∈ `{approve, edit, reject, abort, snooze, ack}`. The inbox path keeps its existing `inbox:*` prefix unchanged — `cockpit:*` is the new namespace. Router is a single `match data.split_once(':')` that fans to `inbox::*` or `cockpit::*` handlers. Documented in ADR 0010 (Task 3).

6. **Per-user authorization via paired-chat allowlist.** New SQLite table `telegram_pairings { chat_id INTEGER PRIMARY KEY, user_label TEXT, paired_at INTEGER, revoked_at INTEGER NULL }`. Every incoming callback **and** every incoming message has its `chat_id` checked against this table before any handler runs. Unknown chats produce a single INFO log (`target: "telegram::auth"` — `admission_denied chat_id={id}`) and the bot replies with a generic "This bot is not paired with this chat" so it does not silently swallow. The pairing flow uses an ephemeral binding token (Task 8): `surge telegram setup` prints a 6-char base32 token; the operator sends `/pair <token>` to the bot from their personal chat; the daemon's command handler looks up the token in `telegram_pairing_tokens` (separate table — see Task 9), enforces a 10-minute TTL and single-use, and inserts a row into `telegram_pairings`.

7. **Edit-feedback ergonomics: operator taps `✏ Edit`, bot sends a follow-up message "Reply to this message with feedback for stage X", uses Telegram's `ForceReply` markup, then the next text message from that chat with `reply_to_message.message_id == <our forced-reply prompt>` is interpreted as the feedback payload.** The forced-reply prompt's `message_id` is stored on the `telegram_cards` row in column `pending_edit_prompt_message_id`. When the reply lands, the daemon: (a) calls `Engine::resolve_human_input(run_id, call_id, json!({ "outcome": "edit", "comment": <feedback text> }))`, (b) clears `pending_edit_prompt_message_id`, (c) edits the original card via `editMessageText` to show "✏ Edit requested — applied". Rationale: matches Telegram-native UX and does not require any state machine outside the cards table; the operator can also send the feedback as a `/feedback <run_id> <text>` command as an explicit alternative (Task 16).

8. **Card update via `editMessageText` is the only update path. Sending a fresh card on update is forbidden.** The cards table stores `chat_id, message_id, last_edit_seq, content_hash`. The cockpit emitter computes a fresh `content_hash` from the rendered body+keyboard and only calls `editMessageText` when the hash changed (otherwise the call is a no-op — saves rate-limit budget). On daemon restart (Task 21), the cockpit reads the cards table and re-derives the card content from the **current** run state; if `content_hash` mismatch ⇒ a single `editMessageText` corrects the card. Never `sendMessage` a new card on resume. Documented in ADR 0011 (Task 4).

9. **Rate-limit handling: token-bucket per chat-id, 1 update/sec sustained, burst 5, plus global ceiling of 25 updates/sec.** Telegram's documented soft limits are 1 msg/sec/chat and ~30/sec across all chats; we leave a safety margin of 5/sec at the global level. Implementation lives in `surge-telegram::rate_limiter::TokenBucket` (uses `tokio::sync::Mutex<TokenBucketState>` keyed by chat-id; global ceiling is a single `Semaphore::new(25)`). Bot API responses with HTTP 429 + `Retry-After` honor the header verbatim and back off. Failed sends are **NOT** retried inline — the cockpit emitter schedules a deferred retry via the existing event-tail watcher so transient 429s do not block the bootstrap stage. Documented in Task 18.

10. **Recovery model: cockpit is stateless beyond the SQLite `telegram_cards` table.** On daemon startup, `surge-telegram::recover::reconcile_open_cards` (Task 21) reads all rows where `closed_at IS NULL`, joins against the run-event log to derive the current state of each gate, and either (a) calls `editMessageText` to refresh the card content (most common — daemon restart between an emission and a response), or (b) marks the card as `closed_at = now` if the underlying gate has already resolved while the daemon was down. **No** new cards are ever sent on resume. Idempotent re-emission requirement (Task 21 + Decision 4 + Decision 8) is satisfied because we never duplicate a card for the same `(run_id, node_key, attempt_index)`.

11. **Event-to-card mapping table (locked here so each task references one row):**

    | Event payload (in `EventPayload`) | Cockpit action | Card kind |
    |---|---|---|
    | `HumanInputRequested { node, summary, options, allow_freetext }` | Send card (or edit if exists) | `human_gate` — approve/edit-reject buttons |
    | `BootstrapApprovalRequested { stage, channel }` | Send stage-aware card preview | `bootstrap_<stage>` — approve/edit/redo |
    | `BootstrapApprovalDecided { stage, decision, comment }` | `editMessageText` to "decided" state, no buttons | finalize same card |
    | `RunCompleted { outcome, .. }` | Emit completion card (new) OR edit progress card | `completion` |
    | `RunFailed { stage, reason }` (or terminal `OutcomeReported` with rejected) | Emit reject card | `failure` |
    | `EscalationRequested { stage, reason }` | Emit escalation card | `escalation` |
    | `PipelineMaterialized { graph_hash, .. }` | Edit bootstrap-Flow card to "pipeline ready" | finalize |
    | `StageEntered { node }` | Update **status** card via `editMessageText` (no spam) | `status` (singleton per run) |

    The cockpit subscribes to a single broadcast channel `RunEventTap` (Task 6) that the engine already owns (verified via `engine.rs:683`-area facade — currently only the persistence writer is downstream; we add a tap subscriber). Mapping enforced by a single dispatch table in `surge-telegram::cockpit::dispatch.rs`.

12. **`HumanInputResolved` event vs the Engine's `resolve_human_input` method.** The Telegram callback handler converts the inline-button tap into `Engine::resolve_human_input(run_id, Some(call_id), response_json)` — the **identical** call console-approval makes from `surge-cli/src/commands/bootstrap.rs:render_console_prompt`. The engine emits `HumanInputResolved` itself; the cockpit does not synthesize that event. Mandatory acceptance: `cargo grep` after implementation MUST find exactly one production call site emitting `HumanInputResolved` (in the engine), and exactly two call sites of `Engine::resolve_human_input` (console + Telegram). Documented as a regression assertion in Task 25.

13. **`/run`, `/status`, `/abort`, `/runs`, `/snooze`, `/pair`, `/feedback` commands** are dispatched by `teloxide`'s `BotCommand` derive. Read-only commands (`/status`, `/runs`) query the event log via `RunEventQuery` (new read API on `surge-persistence` — Task 12). Mutating commands (`/abort`, `/snooze`, `/feedback`) call into orchestrator/engine handles. The command parser lives in `surge-telegram::commands::*` with one module per command. `/pair` is the only command available to *unpaired* chats — every other command is gated on the allowlist (Decision 6).

14. **Inline-button stale-tap handling (24h+ requirement).** The card record stores `closed_at`. If a callback arrives for a card whose `closed_at IS NOT NULL`, the bot replies with `answerCallbackQuery { text: "This card is no longer active", show_alert: false }` and updates nothing. If the card row is missing entirely (DB pruned, very old card), same response with text "Card expired". No panics, no errors propagated to the user. Asserted by Task 24.

15. **Snooze ergonomics from inbox extended to cockpit cards.** `/snooze <duration>` on a card-reply context (operator replies to one of our cards with `/snooze 1h`) inserts a row into the existing `inbox_snoozes` table generalized to a polymorphic `subject_kind` column (`inbox_ticket` | `cockpit_card`). The existing inbox snooze consumer in `surge-daemon::inbox::consumer` already polls and re-emits on expiry — we extend its scan to also re-emit cockpit cards (Task 19). Schema migration is additive (Task 11).

16. **Logging targets** follow the existing `engine::*` / `notify::*` / `cli::*` naming convention: cockpit uses `telegram::cockpit`, `telegram::callback`, `telegram::cmd::<name>`, `telegram::auth`, `telegram::rate`, `telegram::cards`, `telegram::recover`. Levels: DEBUG on routing decisions, INFO on user-visible state transitions (card sent / edited / closed; admission accepted/denied), WARN on rate-limit hits and degraded edits, ERROR before bubbling a `TelegramCockpitError`.

17. **Errors:** new `TelegramCockpitError` enum in `surge-telegram::error` (`#[non_exhaustive]`) — variants `Auth(ChatId)`, `PairingTokenInvalid`, `PairingTokenExpired`, `CardNotFound`, `CardClosed`, `RateLimited`, `Transport(String)`, `EngineResolve(EngineError)`, `Persistence(String)`. The cockpit's outer event loop logs and continues — errors never abort the bot. Telegrammed back to the operator only for explicit user actions (a pairing failure visibly fails the `/pair` command; transient transport errors are logged but invisible).

18. **Clippy budget:** repo-wide `clippy.toml` enforces cognitive-complexity ≤ 25, function length ≤ 100 lines, max 3 bool struct fields, max 7 fn args (3 bool args max). New modules respect these — break long handlers into helpers. The card-dispatch table is the most at-risk; we structure it as one `dispatch_<event_kind>` fn per event variant rather than a single monolith.

19. **No re-architecting `surge-notify`.** The existing `TelegramDeliverer` keeps its push-style `deliver(channel, rendered)` API and continues to handle plain-text notify events. The cockpit owns its own Bot client (separate teloxide handle inside `surge-telegram`) so we do not have to retrofit a callback receiver onto `TelegramDeliverer`. This is the boundary that lets us ship without breaking the existing inbox notification path.

20. **Secrets:** the bot token and admin chat-id are loaded via `surge-persistence::secrets` (existing) keyed under `telegram.cockpit.bot_token` / `telegram.cockpit.admin_chat_id`. `surge telegram setup` populates these. The existing `TelegramSecretResolver` interface continues to back `TelegramDeliverer`; cockpit uses a separate resolver implementation under `surge-telegram::secrets`.

## Risks & Open Questions (Defer or Decide?)

- **Webhook receiver (`tiny_http`).** **Defer.** Roadmap line 180 lists it as opt-in; long-poll is the default and satisfies the milestone's acceptance criteria. The webhook path is a follow-up issue; the cockpit module is structured so that swapping the update source from `polling_default` to a `tiny_http`-backed `UpdateListener` is a single seam.
- **Sandbox-elevation card** (roadmap line 176). **Defer / Out of Scope** — sandbox elevation events do not exist yet; the sandbox-delegation-matrix milestone owns that surface. Plan explicitly excludes this card.
- **Tracker automation tiers** (the next roadmap milestone). **Out of Scope** — the cockpit is consumed by tracker automation but does not own it.
- **Edit-feedback length cap.** **Decide.** Telegram inbound text messages cap at 4096 chars; we accept up to that and truncate with a marker if a future channel routes longer text. No artificial limit imposed here.
- **Card body length > 4096 chars** (artifact previews). **Decide.** Renderer truncates per artifact to 1500 chars with `… (truncated; full artifact in <runs-root>/<run_id>/artifacts/<hash>)` footer; total body capped at 3800 chars to leave keyboard headroom. Implemented in Task 13.
- **Tap-channel buffer 1024.** **Decide.** Sized so a busy run (one event per ~100ms) gives the cockpit a ~100s grace before Lag. Lag triggers `reconcile_open_cards`, so even under pressure the user experience is "card is briefly out of date, then snaps to current"; no card is lost.

## Out of Scope (explicitly)

- Webhook receiver via `tiny_http` (long-poll only for this milestone).
- Sandbox elevation card (owned by the sandbox-delegation-matrix milestone).
- Tracker automation tiers L0–L3 (next milestone in the roadmap).
- Metrics / Prometheus exporter for cockpit counters (tracing INFO is the only observability we ship; counters are derivable from log scrapes).
- Multi-bot or multi-workspace cockpit (single bot, single allowlist this milestone).
- Group-chat support (cockpit assumes 1:1 chats — group `chat_id`s land in the allowlist but no UX guarantees for >1 admin per chat).
- Internationalization (English-only strings).
