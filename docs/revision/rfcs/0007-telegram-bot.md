# RFC-0007 · Telegram Bot

## Overview

The Telegram bot is the primary approval channel for vibe-flow runs. It implements the "describe and walk away" experience by delivering inline-keyboard approval cards on the user's phone, allowing them to make decisions in seconds without a computer.

This document specifies:
- Bot architecture and lifecycle
- Message types and formatting
- Inline keyboard UX
- Slash commands
- Setup and authentication
- Error handling and retries

## Architecture

### Components

```
┌─────────────────────────────────────────────────┐
│ vibe-flow daemon (per-run subprocess)           │
│  ↓ "approval needed" event                       │
│                                                  │
│ vibe-flow telegram service (singleton)           │
│  ├─ Outgoing: format card, send via Bot API      │
│  └─ Incoming: webhook or long-poll for replies   │
│       ↓ user tapped button                       │
│  ↓ "approval decided" event                      │
│                                                  │
│ vibe-flow daemon                                 │
└─────────────────────────────────────────────────┘
```

The Telegram service is a singleton — one process, handles all runs across all projects. It connects bot identity to user identity via configured `chat_id`. The service consumes a queue of outbound messages and produces approval-decision events.

### Why singleton

- One Telegram bot instance can serve multiple runs concurrently
- Webhooks/polling consume one connection; sharing reduces resource usage
- Runs come and go (subprocesses); bot persists across them

### Communication between daemon and bot service

Local IPC via SQLite event log:

- Daemon writes `ApprovalRequested` event with `channel = telegram`
- Bot service polls (or trigger-listens) for new approval requests, sends to Telegram
- User responds, bot service writes `ApprovalDecided` event back
- Daemon resumes when it observes the decision event

This avoids needing a network connection between bot service and daemon — they share the same SQLite database.

## Setup flow

First-time bot connection (one-time per user):

1. User runs `vibe telegram setup`
2. CLI shows a URL: `https://t.me/<your-bot-name>?start=<ephemeral-token>`
3. User opens URL on phone, taps "Start"
4. Bot receives `/start <token>` and binds the user's chat_id to the token
5. CLI polls for binding completion, stores `chat_id` in `~/.vibe/config.toml`
6. CLI displays "Connected — chat ID 12345" and exits

After this, all approvals route to that chat_id by default.

### Token rotation

The bot's API token is read from environment variable `VIBE_TELEGRAM_BOT_TOKEN` or from `~/.vibe/secrets.toml` (mode 0600). Rotation: update the secret, restart bot service.

### Multiple users

For a single user, single chat_id — sufficient for v1. For shared projects (future), configurable per-project chat IDs:

```toml
# ~/.vibe/config.toml
default_chat_id = 123456789

[[chat_overrides]]
project_path = "/work/teamproject"
chat_id = -987654321  # group chat
```

## Bot framework choice

Use **`teloxide`** (Rust-native Telegram bot framework). Reasons:
- Mature, well-maintained
- Async/Tokio idiomatic
- Type-safe handlers
- Webhook + long-poll support
- Inline keyboard well supported

Alternatives considered: `frankenstein` (lower-level, more boilerplate), `tbot` (less active).

## Message types

### Type 1: Run started notification

Sent once at run start, after bootstrap begins.

```
[amber accent line]

▸ Run #0083 started

project    sample-app
pipeline   rust-crate-tdd
stages     7 · ETA ~22 min
sandbox    workspace + crates.io

[==·····················] 0 / 7 stages

[Open dashboard ↗]
```

Inline keyboard: `Open dashboard` (URL button → localhost web view).

### Type 2: Bootstrap approval card

One per bootstrap stage. Fields per stage are:

**Description card:**
- Title: "Description ready · 1 / 3"
- Body: blockquote with summary, key fields (stack, target)
- Buttons: `✓ Approve` / `✎ Edit` / `✕ Reject` / `View full description ↗`

**Roadmap card:**
- Title: "Roadmap ready · 2 / 3"
- Body: list of milestones with task counts and complexity
- Buttons: `✓ Approve` / `✎ Edit` / `✕ Reject`

**Flow card:**
- Title: "Flow ready · 3 / 3 — last gate before run"
- Body: text mini-diagram of structure (loops collapsed), key parameters (sandbox, ETA, cost estimate)
- Buttons: `✓ Approve & start` / `✎ Edit` / `✕ Reject`

### Type 3: HumanGate approval card

Generated by HumanGate nodes during pipeline execution. Content depends on the node's `summary` template.

### Type 4: Sandbox elevation card

Triggered when agent needs sandbox elevation:

```
[yellow accent line]

⚠ Sandbox elevation needed

run        #0083
stage      Implement · 4 / 7

Agent wants to add dependencies that aren't in the rust-crate-tdd allowlist:

$ cargo add nom@7.1
$ cargo add proptest@1.5 --dev

reason       parser combinators · listed in plan.md
trust        crates.io · verified
downloads    nom 89M · proptest 12M

[ ✓ Allow once ] [ + Allow & remember ]
[ ✕ Deny ] [ View plan.md ↗ ]
```

### Type 5: Stage progress (passive update)

Sent every N stages or on important transitions. Edits the prior progress message rather than spamming new messages:

```
[blue accent]

⊙ Stage progress

[████████······] 4 / 7

descr ✓  roadmap ✓  plan ✓  implement ◀  verify  review  pr
```

The progress card is `edit_message`'d in place — single message updates. User scrolls back to see latest state without spam.

### Type 6: Run completion

```
[green accent]

✓ Run #0083 complete

7 / 7 stages · 24m 14s · $4.82
[==========================]

DELIVERED
files       12 created · 0 modified
tests       24 passing · 0 failing
coverage    87.4%
clippy      0 warnings
docs        all public items

+1,847 lines  −0  12 files

[GitHub icon] PR #47 · ready to merge
              github.com/user/json5-serde/pull/47

[ ✓ Merge PR ] [ Open in browser ↗ ]
[ View diff ] [ Run details ]
[ ↻ Re-run from stage ]
```

### Type 7: Run failed

```
[red accent]

✗ Run #0083 failed

stage     Verify · 5 / 7
reason    cargo test failed: 2 of 24 tests failing
elapsed   18m 22s

LAST OUTPUT
test parser::tests::trailing_comma ... FAILED
test parser::tests::nested_objects ... FAILED

[ ↻ Retry stage ] [ ⌖ Replan from milestone ]
[ View logs ↗ ] [ Abort run ]
```

## Inline keyboard mechanics

### Button design

Each button has:
- Visible label
- `callback_data` payload (JSON-encoded)
- Optional URL (for external links — uses `url` instead of `callback_data`)

Callback data format:

```json
{
  "v": 1,                     // schema version
  "rid": "0083",              // run ID (short)
  "ev": 412,                  // event seq triggering this approval
  "act": "approve"            // action ID
}
```

Keep under 64 bytes (Telegram limit).

### Button styles

Telegram doesn't natively support styled buttons (no colors), so semantic differentiation uses:
- Emoji prefix: `✓` (positive), `✕` (negative), `⚠` (caution), `↻` (action), `↗` (external)
- Position: positive actions first, destructive last
- Layouts: 2-3 buttons per row maximum

### Disabled buttons

Telegram doesn't support disabled buttons. Use:
- Show only available actions (don't show buttons that aren't valid yet)
- After click, edit message to show "✓ approved by you" instead of buttons

### After-decision rendering

When user taps a button:
1. Bot service receives `callback_query`
2. Writes `ApprovalDecided` event
3. Edits original message: removes inline keyboard, appends a brief confirmation line
4. Original message stays in chat as audit trail

Example after approval:
```
[original card content]

✓ approved by you · 14:32
```

## Slash commands

The bot supports a small set of slash commands for ad-hoc operations:

```
/start [token]    — initial setup with binding token
/run <description>  — start a new run from Telegram
/list              — show recent runs (last 10)
/status <run_id>   — current state of a run
/cancel <run_id>   — abort a running run (asks confirmation)
/replay <run_id>   — start a replay of a finished run
/help              — list commands
```

### `/run` from Telegram

Initiates a run from Telegram chat:
1. User: `/run add CLI flag for verbose mode`
2. Bot: "Starting run #0084 in current project (sample-app)..."
3. Engine starts daemon, run begins
4. Description Author proceeds, bot will send approval cards as usual

The "current project" is the project most recently used (tracked in `~/.vibe/state.toml`). To run in a different project, user can specify:

```
/run --project /path/to/other "fix bug in parser"
```

### `/list`

Returns up to 10 most recent runs as a numbered list with status emoji:

```
1. 🟢 #0083 sample-app · "Build JSON5 parser" · 24m · $4.82
2. 🟡 #0082 sample-app · "Refactor lexer" · running · 11m
3. 🔴 #0081 myproject · "Add OAuth" · failed at Verify · 8m
4. 🟢 #0080 myproject · "Update deps" · 3m · $0.42
...

Tap a number for details.
```

Each line is a button (callback_data: `{"act":"detail","rid":"0083"}`) leading to a status card.

## Free-text replies

When user replies with free text (no slash command):

- If replying to an approval card: text is captured as the `human_comment` for the corresponding `ApprovalDecided`. Useful for "Edit" path: user tapped Edit, then types feedback.
- If not in any approval context: bot responds with help message ("Use /run, /list, or tap buttons on cards")

The bot does NOT have an LLM-based conversational interface. It's a structured interaction — buttons + slash commands. This keeps it simple and predictable.

## Long-poll vs webhook

For local-first single-user setup, **long-poll** is recommended:
- No public IP required
- No HTTPS certificate
- Simpler to start

For users running on a server with public IP:
- Webhook recommended (more efficient)
- Configurable in `~/.vibe/config.toml`:
  ```toml
  [telegram]
  mode = "webhook"
  webhook_url = "https://my-domain.com/vibe/tg-webhook"
  webhook_secret_token = "..."  # for request validation
  ```

Default: long-poll.

## Error handling

### Telegram API failures

If Bot API returns error (network, rate limit, server error):
1. Engine retries with exponential backoff (max 5 attempts over 5 minutes)
2. After exhaustion, fall back to next channel in priority list (per RFC-0006)
3. If all channels exhausted, run pauses with status "approval undeliverable"; CLI shows warning

### User doesn't respond

Default: wait indefinitely. Run sits as daemon until user taps button.

Configurable timeout per node:
```toml
[node.human_gate.timeout]
duration_hours = 24
on_timeout = "escalate"   # | "reject" | "auto-approve" (rare)
```

### Duplicate notifications

Idempotency: each approval card has a unique key `(run_id, event_seq)`. If bot crashes after sending but before recording, on restart it sends to find unresponded approvals. To avoid double-notification:

- Bot first checks if message already exists in chat (Telegram returns the previous message ID if message_id was tracked)
- Compares hash of intended content
- Only sends if no matching message found

## Multi-run handling

Multiple concurrent runs send separate cards to same chat. To distinguish:

- Each card has run_id badge in title (`Run #0083`, `Run #0084`)
- Cards from different runs interleave by time
- Optional: per-run topic in supergroup (Telegram feature for organizing chats)

For users who feel overwhelmed: `/mute <run_id>` silences notifications from a specific run (still happens but doesn't ping). `/unmute <run_id>` re-enables.

## Privacy and security

### What goes through Telegram

- Run metadata (project name, run ID, status)
- Stage outputs as summaries (description.md content, roadmap structure)
- Diff statistics (counts, not full diffs)
- Cost figures
- Sandbox elevation context (commands being run)

### What does NOT go through Telegram

- Full source code (only stats)
- API keys, tokens, secrets (filtered before sending)
- Full file contents (link to local web view instead)

Users concerned about Telegram seeing their code: it's a local approval system, not a code review tool. Code stays local; Telegram sees abstractions.

For corporate / sensitive contexts: use Slack instead (RFC-0006 mentions other channels).

### Secret filtering

Before sending any message, content is scanned for:
- API key patterns (long random strings near keywords like "key", "token", "secret")
- Stack traces with file paths (paths preserved, but suspicious-looking strings redacted)

Filtering is best-effort. Users should not embed secrets in approval-relevant artifacts; AGENTS.md system prompts should instruct agents to elide secrets in summaries.

## Bot identity

The bot displays as `vibe·flow` with profile image (custom avatar shipped with project). Bot username is unique per user (e.g., `vibe_flow_<user_initials>_bot`) to avoid conflicts.

For multi-user installations on shared servers: each user creates their own bot via @BotFather, gets unique token.

## Acceptance criteria

The Telegram bot is correctly implemented when:

1. `vibe telegram setup` completes binding within 30 seconds.
2. Bootstrap approval cards arrive within 5 seconds of stage completion.
3. Inline button taps result in `ApprovalDecided` event within 2 seconds.
4. Slash commands `/run`, `/list`, `/status`, `/cancel`, `/replay`, `/help` all work as specified.
5. Bot service restart does not lose pending approvals (event-sourced model recovers).
6. Long-poll consumes <10MB memory for idle bot service.
7. Multiple concurrent runs (3+) send distinguishable approval cards without confusion.
8. Free-text reply during "Edit" approval flow correctly captures user feedback as artifact.
9. Webhook mode works correctly when configured (integration test).
10. Secret filtering catches at least 90% of common API key patterns in test fixtures.
